# v1.0 Milestone Audit Report

**Milestone:** git-worktree-tools CLI Consistency
**Audited:** 2026-03-09
**Status:** ⚠ gaps_found
**Score:** 17/20 requirements satisfied

## Executive Summary

Six phases of refactoring work successfully wired the shared logger, UI primitives,
interactive menu fixes, JSON output coverage, in-process delegation, and legacy CLI
wiring completeness. Integration checking uncovered two JSON-mode code paths missed
during Phase 4 verification: `wt prs --json` does not call `setJsonMode`, and
`wt new --json` (interactive mode) emits bare `console.log` output before the JSON
envelope. These break the LLM-01 contract and trigger the FAIL gate.

### Integration Gaps

**INT-A: `wt/prs.ts` missing `setJsonMode(!!argv.json)`** (LLM-01, UNI-03)

- Every other `wt/*` handler calls `setJsonMode` before delegating
- `wt/prs.ts` invokes `runPrsCommand()` without setting the mode gate
- The `runPrsCommand` internal JSON handling exists, but shared `print()` calls in the prs code path bypass the JSON suppression

**INT-B: `newpr.ts handleScenario()` bare console.log in JSON mode** (LLM-01)

- Lines 183-184, 243-244, 258-270: scenario description text via raw `console.log()`
- `wt new "desc" --json` without `-y` hits these paths before the JSON envelope
- Produces mixed stdout: description text + JSON result

**INT-C: `src/cli/wt/run-command.ts` orphaned dead code** (no requirement)

- File remains after Phase 5 direct-call migration replaced all `runSubcommand` usage
- No functional impact; structural debt only

   Fix: guard lines 183-184, 243-244, 258-270 with `if (!isJsonMode())` or migrate to `print()`
